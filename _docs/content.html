Abstract

Want to submit an awesome project? Great! But, you need to make sure that your content adheres to our content policy. Let's learn about it in this article!

Student Server Space, first and foremost, is a testing ground for students to get their projects up and running. It's designed for everybody to use it in good spirit, but we need a set of guidelines to clarify what is allowed and what's not. Generally, here's what we're looking for in our submissions:

  • Open-sourced software
  • Non-commercial use
  • Content-Controlled
  • Equitable and Legal

What do those mean exactly? We'll explain in these next few sections.

Overview

Very quickly, here's a table that outlines how our requirements for any submissions. You can find more information about each table category on the page categories, as well as examples and guides.

Licensing

Spiritually, Student Server Space is about education through community. Or, as My Little Pony puts it, "sharing is caring". Because of that, and also because of just how awesome open-source is, we require all submitted projects (and all deployed projects) to be open-sourced, and have an open-source license to account for that.

Definition

An Open-source license is a type of license for computer software and other products that allows the source code, blueprint or design to be used, modified and/or shared under defined terms and conditions.

-Wikipedia

Your license should be clearly stated somewhere in your project, either on the project's website, or in the root directory of the project (as a LICENSE, LICENSE.md, or LICENSE.txt file).

We're not going to tell you what specific license you need to choose, but there are a few common ones that we'd recommend (specifically the MIT and Apache Licenses). If you're interested about learning more about licenses in general, try out choosealicense.com.

Fun Fact

Student Server Space's website is under the MIT License. You can check it out here.

Not having a License will mean we will deny your project, and likewise having a closed-source license will also get your project denied. If you have any more questions about licensing, feel free to ask us!

Commercial Policy

Student Server Space is here to get developers up and running on their side projects, but we aren't here to provide a mechanism to cut costs on student business. While we do like the idea of commercial projects, our infrastructure isn't designed for commercial projects. Therefore, we do not allow commerical projects on Student Server Space.

Definition

We define commercial projects as any project that is designed to sell a product, service, or perform monetary transactions.

That being said, we do allow websites that display information about a product, but does not perform the monetary transaction on the site (e.g. mailing in a cheque). In addition, we also make a special exception for charity websites. If you have a question about our commercial policy, let us know: we are more flexible on this guideline, than, let's say, licensing.

Content Control

If your site collects data and/or displays data, there are a few sets of controls that we have to place on your data. Here's a table outlining them all:

Type Explanation
Content Display SSS Projects cannot have public-facing content that is user-editable. This includes anything a user of your website can see on your website, including messages, pictures, and code, from other parties. Private content can still be user-editable: a user can edit content that they themselves can see, but other users cannot.
Content Upload Users cannot upload files to the deployed webserver, period. If your project is deployed elsewhere (not on SSS), then users can upload content: but ramifications that arise from user uploaded content cannot be held accountable to Student Server Space.
Data Consent SSS Projects must make it clear what information they are storing. For example, if an email is required to register, then this does not need to be clarified further; on the contrary, if the application stores the user's IP, then this must be stated on the website.
Emails Deployed projects cannot use internal mail servers. In addition, any emails sent out by your project must be done so after the user has consented to receiving these emails (e.g. "I allow X to send me emails" checkbox). An exception is made for account creation emails, and account security emails.
Account Deletion If your project features an account system of any kind, there must be a mechanism that allows the user to delete the account fully from the system, with all related information removed. This could be a "delete account" button, an email to contact, etc.

Equity and Legality

This section should be obvious, but your content should be equitable and legal. If it isn't we'll terminate all of your projects, and a warning; a repeat offense is an indefinite ban from the system. There isn't much else to it. Don't be unequitable, don't do illegal things.

Definition

Equitable content is content that's inclusive for all. That means no discrimnation, hate speech, or treating any group of people unfairly. Need help figuring out whether or not your content is equitable? Ask an SSS member!

Definition

Legal content is content that abides to Canadian law. Examples of illegal content includes selling illicit materials, distributing child pornography, or inciting hate speech.

If you need any clarification for this section, even though it is pretty straightforward, please let us know!
