Abstract

Want to submit an awesome project? Great! But, you need to make sure that your content adheres to our content policy. Let's learn about it in this article!

Student Server Space, first and foremost, is a testing ground for students to get their projects up and running. It's designed for everybody to use it in good spirit, but we need a set of guidelines to clarify what is allowed and what's not. Generally, here's what we're looking for in our submissions:

  • Open-sourced software
  • Non-commercial use
  • Well-designed, well-maintained, and secure code
  • Easily deployable
  • Content-Controlled
  • Equitable and Legal

What do those mean exactly? We'll explain in these next few sections.

Overview

Very quickly, here's a table that outlines how our requirements for any submissions. You can find more information about each table category on the page categories, as well as examples and guides.

Source Code Submitted code must be on a Git-based Open-Source platform; we have be able to git clone YOUR-REPOSITORY.git to deploy your code. GitHub
Licensing Submitted Code must have an Open-Source License present in the source code, in a clearly-defined LICENSE file or similar. Apache License or MIT License

While we won't outright deny your code on this factor alone, it will play into whether or not it is accepted. There are resources in the "code design" section of this guide to help you out. Properly named variables, alt attributes in <img> tags, commented code.

Source Code

All of your project's code should be open-source. According to Wikipedia, open-source is defined as:

... computer software with its source code made available with a license in which the copyright holder provides the rights to study, change, and distribute the software to anyone and for any purpose.
Reasoning

We require open-source software for two reasons. Firstly, we believe in principle that open-source software is beneficial: you're giving back to the community, allowing other people to learn from your work, and foster collaborative development. Open-source is becoming more and more popular in the workforce, and we want to make the transition easy for you. The second reason is that it makes integration very easy for us: we can view all of your source code easily, and deploy it even easier.

For simplicty, we also require your code to use a Git-based implemenation of open-source. Git makes version control very easy, and also makes your own code cleaner and manageable.

GitHub

Want some examples of ways to go open-source, with Git? Our main recommendation is GitHub, which is a free, git-based, open-source website and program. We use GitHub for this website, as well as the majority of our projects. GitHub is very easy to use, has an amazing amount of resources, and a great community to help you get started. If you want to learn more about Git and GitHub, check out our page on learning to use Git.
