Overview

Notice

This article tells you how to submit your project to the Student Server Space, not how to deploy it. To deploy your code, read our Deployment Guide, after submitting your project.

To clarify, submitted projects do not need to be deployed (they can be hosted externally), but all deployed projects must be submitted.

Want to submit your project to us? Sounds good! Here's an article that outlines exactly how you would do that. Let's get started.

Prerequisites

Before you try to submit your project, make sure that you've checked off everything in our Content Policy. In addition, if you're also looking to deploy your project on Student Server Space, make sure it adheres to our Code Policy.

In addition, you'll also need these tools to submit your project:

  • A GitHub Account
  • Git, or a Git Client
  • A Text Editor (we recommend Atom or Sublime Text)

Setup

Let's setup your local development machine so you can make some changes! The first thing you're going to do is fork and clone the Student Server Space Website repository on GitHub, so you can work on it. Here's a photo that shows you what button to press:

Now, you should have your own copy of the website repository. We're going to clone that to our computer. You can either do this from the command line, or through a Git client. Regardless, you'll need to press the "clone or download" button, represented in this image: with git clone YOUR-REPOSITORY-URL.git, or through a Git client, like GitHub Desktop. Here's a visual representation of how to open it in GitHub Desktop, from the GitHub website:

If you want to clone it from the command line, type in git clone YOUR-REPOSITORY-URL.git in your terminal, with YOUR-REPOSITORY-URL.git being the link in that picture. If you'd like to open it in GitHub Desktop, simply press the "Open in Desktop" button.

Warning

Make sure you're cloning your fork (version) of the website repository, YOUR-GITHUB-USERNAME/website, not the one from the studentserverspace group. You can't edit that one!

Now you should be ready to make some changes! From now on, we'll be working in the repository directory (wherever you cloned the repository). Get your text editor out, and let's get started!

Adding Projects

Adding your project to the Student Server Space directory requires you to make one (or two) files. First, we're going to make a new YOUR-PROJECT-NAME.md (markdown) file in the _projects directory. Then, we're going to add our YAML header: this is something specific to Jekyll, if you're interested. Here's an example template, for Paleta. 

      
        ---
        layout: projects
        title: Paleta
        short: paleta
        permalink: /projects/paleta/
        github: "malsf21/paleta"
        website: "https://malsf21.github.io/paleta/"
        img: "paleta-1.png"
        creator: matthewwang
        description: "Paleta is a set of style guides for UCC organisations and projects to make design easy."
        ---

Confused by what those categories mean? No problem! Here's a nifty table explaining each element:

Element Name Element Description Required
layout This is a Jekyll-related element: just leave it as projects
title The HTML title of your page. We suggest you make it your project name.
short This is kind of like a slug for your project title. Make it something that's one word, all lowercase.
permalink This is the URL of your project on the directory. It should always be /projects/YOUR-PROJECT-URL. If your project name includes characters that can't be in a URL, please remove them.
github The GitHub-relative URL of your project. It should be in the form of YOUR-USERNAME/PROJECT-NAME
website If your project has a website, put the full URL here. If you don't have one, don't sweat it! Just omit this parameter.
img An image of your project in action. If you don't have any ideas, just screenshot your project in action! Drop your img file in /img/projects
creator The creator key for your project. More on this below.
description A short, one-line description of what your project does.
Warning

Make sure your project has all the required header elements. If it doesn't, we'll deny your submission, no exceptions. You can always resubmit after making corrections.

Once you're done your header, the content of the .md file is a full description of your project. Make sure it's somewhat descriptive, since you're showcasing your project to everybody! Copying your GitHub README.md file is perfectly fine, as long as it's still descriptive. And remember, this is a public-facing site: ensure your description is appropriate for all ages!

Adding Creators

Tip

If you've already added yourself as a creator before, you don't need to repeat this step.

At Student Server Space, we care a lot about attributing creators! When you submit your project, you have the ability to add your creator information.

To add a new author, navigate to /_data/creators.yml. You'll have something that looks like this: 

      
        matthewwang:
          name: Matthew Wang
          site: https://matthewwang.me
          bio: Matt is in love with coding, debate, GitHub, burritos, and mediocre music. Matt spearheaded this project, and handles content, front-end design, and documentation.
          github: malsf21

        jacksarick:
          name: Jack Sarick
          site: https://sarick.tech
          bio: Jack is an IB2 student who builds amazing things with code, english, metal, and more. Jack handles back-end design and server administration.
          github: jacksarick

To add a new entry (you!), all you have to add is a slug key, yourname:, and nest your full name as a string, name: Your Name. Here's a description of all the possible options you can have:

Element Name Element Description Required
name Your full name! If you have a name that includes chars that trip up YAML or Ruby, try encasing it in "", or HTML escape it.
site If you have a website, you can add a link here! Make sure it's a full link, and make sure it's appropriate.
bio A short, one or two line bio. Keep it short and sweet for best effects!
github Your GitHub username, if you have one. Just include the username, no link required.

Once you add your own key to the creators.yml, you're ready to submit your pull request.

Project Testing

Now that you've added your project (and your creator file) to your repository, we're going to test them to see if they work, using Jekyll. You'll need Ruby to use Jekyll, but that comes pre-installed on most unix systems.

Tip

Installing Jekyll is super easy! Just type in gem install jekyll into your command line/terminal.

We're now going to test your new code to see if it works! In the root directory of your project, or /, type in your command line jekyll serve. If your output looks something like this:

Then it's working! If some error messages pop up, see if you can fix them yourself (google is a great resource). After having no compile errors, you can check out the website yourself. Visit the address that comes after Server addrress in your browser, and try checking the Projects Directory to see if your set up works as intended.

Submitting Projects

Now that your project is submitted in your local repository, and tested, we're going to bring it up to speed with our main branch! Here's a graphical explanation of what we're doing in principle; after, we'll walk you through how to do it with Student Server Space.

The first thing we want to do is commit our changes to our remote branch on our fork. In other words, that means pressing that commit + sync button on GitHub Desktop, or doing git commit and git push origin master on your command line. After your fork repository is up to speed with your local development file system (you can double check this by visiting YOUR-USERNAME/website on GitHub), we're going to submit a Pull Request.

A Pull Request sends your new changes, in your forked repository, and asks the main SSS website's repository to merge these changes. In this case, you'll do this through GitHub. Head to the "Pull Requests" section of your fork, and press "New Pull Request" button.

You should get a screen that looks like this:

Press that "Create pull request" button. Now, you should be a screen like this:

Fill out the paragraph description (it'll make things easier on us), and click "Create pull request"! And you're done! Wait until your pull request gets responded to: we'll either merge it, and put it on the website, or deny it, and let you know what you need to fix so it becomes mergeable.

Ta-da! We're done!
